# sygen-updater

The native Sygen updater. Polls GitHub Releases for new `sygen` /
`sygen-admin` versions and exposes a local HTTP endpoint that the core
`/apply` flow calls to atomically swap in a new venv + admin tarball.

This replaces the v1.6 Docker-based updater (which polled GHCR for image
digests and ran `docker compose pull && up -d`). v1.7+ ships native
processes managed by `launchctl` (macOS) or `systemd` (Linux).

## How it works

- Every `CHECK_INTERVAL` seconds the updater:
  1. GETs `api.github.com/repos/alexeymorozua/sygen-releases/releases`
     (the public mirror of release artefacts), filters by `tag_name`
     prefix — `core-` for `sygen` + `sygen-updater` wheels, `admin-` for
     the `sygen-admin` tarball — and picks the highest numeric semver.
  2. Compares against the version pinned in `$SYGEN_HOME/.env`
     (`SYGEN_CORE_VERSION` / `SYGEN_ADMIN_VERSION`).
  3. Writes `{core, admin, checked_at}` to
     `$SYGEN_HOME/host_updates/_updates.json` — core reads it back via
     `GET /api/system/updates`.

  Source repos (`alexeymorozua/sygen`, `alexeymorozua/sygen-admin`) are
  private; only the mirror is reachable to anonymous `curl`. CI in each
  source repo auto-mirrors the release assets on every `vX.Y.Z` tag.
- `GET /health` returns `{ok, last_check, last_error, check_interval,
  venv_dir, admin_dir}` without auth — safe to hit from a smoke test.
- `POST /apply` requires `Authorization: Bearer $SYGEN_UPDATER_TOKEN`.
  It:
  1. resolves latest versions from GitHub Releases,
  2. downloads the new wheel + admin tarball,
  3. creates a new venv, `pip install`s the wheel, `mv`-swaps
     (`venv → venv-prev`, `venv-new → venv`),
  4. extracts the admin tarball, `mv`-swaps the same way,
  5. updates `.env` with the new version pins,
  6. restarts core + admin via `launchctl kickstart -k` /
     `systemctl restart`.
  The updater itself stays alive across the swap (it can't kill its own
  process); a manual `pip install --upgrade sygen-updater` (or a fresh
  `install.sh` run) is the path to update the updater binary.

## Environment variables

| Var | Default | Purpose |
|-----|---------|---------|
| `SYGEN_UPDATER_TOKEN` | *required* | Bearer secret for `/apply` and `/check`. Generated by `install.sh` and persisted in `$SYGEN_HOME/.env`. |
| `SYGEN_HOME` / `SYGEN_ROOT` | platform default | Install root. macOS: `~/.sygen-local`; Linux: `/srv/sygen`. |
| `SYGEN_RELEASES_GITHUB_REPO` | `alexeymorozua/sygen-releases` | Public mirror repo polled with prefixed tags (`core-X.Y.Z`, `admin-X.Y.Z`). Set to an empty string to opt into legacy v-tag mode against the source repos (only useful with `GH_TOKEN`). |
| `SYGEN_CORE_GITHUB_REPO` | `alexeymorozua/sygen` | Source repo for legacy mode (ignored when mirror is set). |
| `SYGEN_ADMIN_GITHUB_REPO` | `alexeymorozua/sygen-admin` | Source repo for legacy mode (ignored when mirror is set). |
| `SYGEN_VENV_DIR` | `$SYGEN_HOME/venv` | Path to the live venv (the swap target). |
| `SYGEN_ADMIN_DIR` | `$SYGEN_HOME/admin` | Path to the live admin dir. |
| `SYGEN_UPDATER_HOST` | `127.0.0.1` | Listen host. Always loopback by default. |
| `SYGEN_UPDATER_PORT` | `8082` | Listen port. |
| `CHECK_INTERVAL` | `1800` | Poll interval in seconds. |
| `STATE_PATH` | `$SYGEN_HOME/host_updates/_updates.json` | Where to write the state JSON. |
| `SYGEN_APPLY_TIMEOUT` | `600` | Hard timeout (seconds) per apply invocation. |
| `GH_TOKEN` | unset | Optional GitHub token for rate-limit relief. |

## Standalone run (for local testing)

```sh
SYGEN_HOME=/tmp/sygen-test \
SYGEN_UPDATER_TOKEN=test \
python3 -m updater
```

```sh
curl -sS http://127.0.0.1:8082/health
curl -sS -X POST http://127.0.0.1:8082/check -H 'Authorization: Bearer test'
```

## Releases

- Push a `vX.Y.Z` tag in this repo → builds a wheel and uploads it as a
  GitHub Release asset. The wheel is consumed by `install.sh` (initial
  install) and by core's `/apply` flow (in-place upgrade).
- The wheel name is `sygen_updater-<X.Y.Z>-py3-none-any.whl`.

See `.github/workflows/release-docker.yml` (legacy, disabled) — there
is no separate release workflow today; the wheel is built as part of
the core repo's `publish.yml` so the version stays in lockstep with
`sygen` itself.
